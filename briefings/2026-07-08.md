### Hackers Can Use 9 of the Most Popular AI Tools to Assemble Massive Botnets

Source: Ars Technica · Published: 2026-07-08 · Read time: ~6 min

**Elevator pitch:** Researchers demonstrate that LLMs hallucinate specific package names with enough consistency that attackers can pre-register those names with malware, and that this vector is already propagating at scale across popular AI coding tools.

**Opinion vs. fact:**
- [FACT] The attack is named "HalluSquatting" (also "slopsquatting" in prior research); it exploits LLMs' tendency to hallucinate plausible-sounding dependency names.
- [FACT] 43% of hallucinated package names recur identically across 10 separate runs of the same prompt; 58% recur more than once.
- [FACT] Researcher Charlie Eriksen (Aikido Security) discovered the hallucinated npm package "react-codeshift" had propagated to 237 repositories via forks before he claimed the name.
- [FACT] A test package registered under the hallucinated name "huggingface-cli" received 30,000 downloads in three months with no active promotion.
- [FACT] The attack applies specifically to AI tools that browse the web or follow documentation links as part of their operation — including Claude Code, Cursor, Devin, and MCP-enabled agents.
- [OPINION] This represents a fundamentally new class of supply-chain risk that existing package-vetting tooling was not designed to detect.

**References:** Aikido Security researcher Charlie Eriksen (react-codeshift discovery, January 2026).

**Justifications:** The quantitative claims are specific, named, and independently verifiable. The 43%/58% persistence rates establish the economic viability of the attack: high enough recurrence to target specific names with confidence. The 30,000-download figure for a passive test package demonstrates the vector is live, not theoretical. The "botnet assembly" framing — the Ars Technica-specific angle — refers to AI tools being used to coordinate malware distribution at scale across the 9 named tools.

**Core points:**
- AI coding agents that browse documentation, fetch READMEs, or install packages on instruction are the primary attack surface
- Hallucinated names are stable: the same LLM generates the same wrong package name repeatedly, making pre-registration a reliable attack setup
- Attack has moved from PoC to evidenced propagation: 237 repo contaminations, 30K downloads from a single test package
- No large-scale confirmed breach yet, but the infrastructure for one is demonstrably active
- Mitigations: package lockfiles, dependency pinning, hallucination-aware supply-chain scanners (Aikido, Snyk); audit what URLs your MCP servers point to

**Impact:** This is immediately actionable for any team building agents that write code, manage dependencies, or follow AI-generated installation instructions. Any agent with write access to a package manifest is a potential vector. MCP server adoption should be audited for what install URLs and tool references those servers expose to the model. CI/CD pipelines using AI-assisted code generation need dependency hygiene review now, not post-incident. The attack's effectiveness also grows with agent autonomy — the more capable and unsupervised the agent, the more dangerous this becomes.

- [FACT] Llama 3.3 8B at Q4_K_M quantisation scores 73.0 on MMLU — performance that previously required GPT-4-class API access.
- [FACT] The primary inference bottleneck in local LLM deployment is memory bandwidth and VRAM capacity, not raw compute throughput.
- [FACT] Google TurboQuant (March 2026) compresses KV cache to 3 bits per value with no reported accuracy loss, reducing KV cache memory 6×.
- [OPINION] Claims about local model feasibility and coding capability had become "too significant to ignore" — Böckeler's stated reason for revisiting the topic.
- [OPINION] Local models are now viable for bounded, structured coding tasks; complex multi-file reasoning still favours cloud frontier models.

**References:** Thoughtworks engineering practice; Llama 3.3 public benchmarks; Google TurboQuant (March 2026).

**Justifications:** Böckeler's practitioner background at Thoughtworks means this is engineering judgment grounded in real delivery friction, not benchmarking in isolation. The MMLU figure for Llama 3.3 is against published benchmarks. The memory bandwidth claim is physics-grounded — well-established in the literature. The TurboQuant figure is from a March 2026 Google release.

**Core points:**
- Viability factors in rough priority: VRAM capacity, memory bandwidth, quantisation fidelity, task-appropriate context length, latency tolerance
- Modern 4-bit quantisation has closed a significant portion of the quality gap vs. API models for structured, bounded tasks
- Memory bandwidth is the real ceiling — adding GPU compute doesn't help when weight-loading throughput is the bottleneck
- Local models remain weaker than cloud frontier models on long-context and multi-file reasoning tasks
- The economic argument requires honest accounting of hardware cost, ops overhead, and team expertise against the gains in latency or data privacy

**Impact:** For a solutions architect deciding where to run inference, this provides a practical framework for a question that is increasingly live. The viability threshold for local models has moved: code completion, small-scope refactors, and documentation generation are now viable targets. Long-context reasoning and complex architectural analysis still favour cloud models. In a telco OSS context, data residency requirements are the clearest trigger for local model investment — and this memo is the kind of grounded assessment that can anchor that build-vs-buy conversation with engineering leadership.

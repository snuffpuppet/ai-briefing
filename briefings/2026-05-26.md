### Harness, Scaffold, and the AI Agent Terms Worth Getting Right

**Source:** Hugging Face Blog · **Published:** 2026-05-25 · **Read time:** ~8 min

**Elevator pitch:** A practical glossary that pins down the definitions of "model," "scaffold," "harness," and "agent" as these terms continue to be used inconsistently across the field — motivated by persistent confusion reported at ICLR 2026.

**Opinion vs. fact**

- [FACT] Claude Code's own documentation describes it as "the agentic harness around Claude."
- [FACT] ICLR 2026 attendees reported genuine confusion about whether "harness" and "scaffold" referred to the same concept or different ones.
- [FACT] A model on its own has no memory between calls and no execution loop — it answers one prompt and stops.
- [OPINION] Harness engineering is becoming the primary site of architectural differentiation as model capabilities converge across vendors.
- [OPINION] The vocabulary confusion is a symptom of an underlying design problem, not just a communication problem.

**References:** Claude Code documentation; ICLR 2026 post-conference discussion. Products cited: Claude Code, OpenAI Codex, Antigravity CLI.

**Justifications:** The article anchors definitions in observable system behavior and in live, shipping products — which makes them harder to dismiss. The strongest evidence is the Claude Code cite: when a major deployed system uses a term explicitly, that is the closest thing to a canonical definition available. The weakest point is that the scaffold/harness boundary does not map cleanly onto all major frameworks (LangGraph and AutoGen both blend the two), so teams using those stacks will need to translate.

**Core points**

- Model = the LLM only. No loop, no tool execution, no persistent memory between calls.
- Scaffold = the behavior-defining layer around the model: system prompt, tool descriptions, response parsing, context management.
- Harness = the execution layer: calls the model, dispatches tool calls, decides when to stop. Everything that is not the model.
- Agent = Model + Harness in the broad-use sense; scaffold is what the harness uses to configure model behavior.
- As model capabilities commoditize, the harness layer is where product-level reliability and differentiation increasingly live.

**Impact:** For a solutions architect designing agentic workflows, the scaffold/harness split is a useful architectural seam. Scaffold concerns — what the model "knows" about its task, what tools exist — belong in configuration or prompts. Harness concerns — how tool calls are executed, when to retry, when to hand off, how to prune context — belong in code with proper observability. If your team treats "agent" as a monolithic concept, you are likely missing that seam and will find reliability issues harder to isolate. For telecom OSS integration specifically, the harness layer is where network API timeouts, retry budgets, and context-pruning decisions belong — getting this decomposition right early avoids expensive refactors later.

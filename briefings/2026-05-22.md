### Bliki: Vibe Coding

**Source:** Martin Fowler · **Published:** 2026-05-21 · **Read time:** ~4 min

**Elevator pitch:** Fowler formally defines vibe coding and establishes a clear scope boundary: it is appropriate only for disposable software with limited audiences, and should not be applied to production systems.

**Opinion vs. fact**

- [FACT] The term "vibe coding" was coined by Andrej Karpathy in February 2025
- [FACT] Vibe coding is defined as building software by prompting an LLM without reading any of the generated code
- [FACT] The technique is accessible to people without programming knowledge
- [OPINION] The resulting software often shows problems with maintainability, correctness, and security (Fowler's practitioner assessment; no empirical study cited)
- [OPINION] It is best suited for disposable software written for a limited, tolerant audience
- [OPINION] Overconfidence in vibe coding heightens the risk of inadvertently introducing vulnerabilities

**References:** Andrej Karpathy (coined the term, Feb 2025); Fowler's related writing: "To vibe or not to vibe," "Harness engineering for coding agent users"

**Justifications:** Fowler does not cite empirical failure-rate studies — this is experienced practitioner judgment. The weakest point is the absence of measured outcomes. The strongest point is logical: if you don't read the code, you cannot verify correctness or reason about the security surface. Fowler's adjacent work on harness engineering provides context for what the correct alternative looks like.

**Core points**

- Definition: build software by prompting an LLM, test the output, prompt for changes — without reading the generated code
- Known failure modes: maintainability, correctness, security
- Appropriate scope: disposable software for a limited, tolerant audience; unsuitable for production systems
- Fowler is implicitly defining what vibe coding is *not* — a contrast to responsible agentic engineering with harnesses and verification
- Entry links to "To vibe or not to vibe" and "Harness engineering" as the positive alternatives

**Impact:** For an architect setting team policy: this bliki gives you precise language for a common conversation. The scope boundary ("disposable, limited-audience software only") is what makes it actionable. Any team using Claude Code or similar on a production codebase should be explicitly above "vibe coding" — maintaining code review, verification steps, and harness engineering practices. Fowler's entry exists in part to give technical leads a reference point for communicating that distinction.

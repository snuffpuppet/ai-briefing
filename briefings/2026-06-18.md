- [OPINION] ARD and MCP are complementary layers (discovery vs. invocation) — framing from spec contributors

**References** — Microsoft, Google, GoDaddy, Hugging Face (specification contributors); Google Developers Blog; GitHub: huggingface/hf-discover

**Justifications** — Multi-stakeholder contributor list is specific and credible. Reference implementation on GitHub is verifiable. The "draft" status is appropriately caveated throughout. The adoption dependency — registries need to implement ARD for it to function — is the key unknown not addressed in the announcement.

**Core points**
- ARD defines how agents query federated registries for capabilities at runtime, rather than requiring pre-installed tool manifests at design time
- Scope covers Skills (reusable agent behaviors), MCP servers, and ML applications
- Reference implementation is a CLI, callable from agent harnesses
- Google + Microsoft + Hugging Face backing is meaningful signal for standards momentum
- Complements MCP: MCP handles tool invocation; ARD handles tool discovery

**Impact** — ARD addresses a real constraint: agents today need to know about tools at design time, limiting runtime composability. If ARD achieves adoption comparable to MCP, it enables agents that can extend themselves at runtime by locating the right capability for a task. For enterprise systems, this also opens new governance surface — policy will be needed to govern which registries agents are permitted to query and what trust level to assign to discovered tools. Worth monitoring closely; the multi-stakeholder backing and reference implementation suggest this is past concept stage.

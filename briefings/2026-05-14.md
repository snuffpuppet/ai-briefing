### 1. Building a safe, effective sandbox to enable Codex on Windows

**Source:** OpenAI News · **Published:** 2026-05-13 · **Read time:** ~8 min

**Elevator pitch:** OpenAI explains the concrete security engineering required to make an
autonomous coding agent safe on Windows — a platform that offers no single "safe subprocess"
primitive.

**Opinion vs. fact**

- [FACT] Codex on Windows provides two sandbox modes: Elevated (dedicated lower-privilege sandbox
  users, filesystem permission boundaries, firewall rules) and Unelevated (restricted Windows
  token, ACL-based filesystem boundaries, environment-level offline controls).
- [FACT] Default configuration denies all network access; any internet use requires explicit
  operator opt-in.
- [FACT] Write access is scoped to the active workspace directory; read access is broader across
  the filesystem.
- [FACT] An agent approvals system lets operators gate individual actions before execution.
- [OPINION] The elevated sandbox is described as "preferred," implying the unelevated fallback
  involves accepted residual risk — the authors' framing, though it reflects documented
  capability differences.
- [OPINION] "Windows did not provide one primitive that cleanly maps to 'safe autonomous coding
  agent.'" — The authors' characterisation of the platform gap; accurate but self-serving.

**References:** OpenAI Codex developer documentation (developers.openai.com/codex/security,
/concepts/sandboxing, /windows, /agent-approvals-security). No external studies cited.

**Justifications:** The strongest point is the concrete two-mode design with explicit statements
of what each mode enforces — this is engineering documentation, not marketing. The weakest point
is the absence of an adversarial analysis: the post describes what the sandbox does, not what it
fails to prevent under realistic attack.

**Core points**

- Windows lacks a unified "safe agent subprocess" primitive; OpenAI composed filesystem ACLs,
  lower-privilege user accounts, and firewall rules to approximate one.
- Elevated mode is the primary path; unelevated is the fallback when elevation cannot be granted
  (e.g., in environments where the user lacks admin rights).
- Network is off by default — any network access is an explicit opt-in that remains visible to
  operators.
- Write scope is limited to the workspace; read scope is broader. This means exfiltration via
  side channels (a tool that reads a credential file and includes it in output) is not
  automatically blocked by write restrictions alone.
- The agent approvals layer adds a human-in-the-loop gate on top of the sandbox — defence in
  depth rather than relying on the sandbox alone.

**Impact:** For anyone designing agentic coding systems, this is a concrete reference
implementation. The architectural lesson generalises beyond Windows: sandboxing an autonomous
agent requires a stack of controls (process isolation + filesystem scope + network policy +
human approval gates), and the composition matters more than any individual control. The
asymmetry between read and write scope is worth noting explicitly when threat-modelling your own
deployments — it is an easy gap to miss.

### 3. [AINews] The End of Finetuning

**Source:** Latent.Space · **Published:** 2026-05-13 · **Read time:** ~8 min

**Elevator pitch:** Fine-tuning is losing its status as the default customisation path for most
teams, displaced by very long system prompts and in-context approaches — though elite teams are
moving in the opposite direction with increased RLFT usage.

**Opinion vs. fact**

- [FACT] Companies including Cursor and Cognition have increased, not decreased, their use of
  open-model reinforcement learning fine-tuning (RLFT).
- [FACT] Very long context prompts (e.g., Anthropic's model spec / Claude's system prompt) are
  documented as a production alternative to fine-tuning for behaviour shaping.
- [FACT] Continued prefill/decode (P/D) disaggregation in inference infrastructure is making
  very long context prompts more economically viable at scale.
- [OPINION] Fine-tuning is characterised as "the next casualty of the 2026 Side Quest massacre"
  — the author's framing that major labs are deprioritising capabilities that don't fit the
  agentic product trajectory.
- [OPINION] The "Finetuner's Fallacy" — early training data leaves a durable imprint that later
  fine-tuning struggles to undo — is introduced as a named concept; appears to be the author's
  own framing rather than an established result.

**References:** Fast.ai's Jeremy Howard cited on continued pre-training as a fine-tuning
alternative. No peer-reviewed studies cited.

**Justifications:** The strongest point is the Cursor/Cognition counterexample, which correctly
prevents an oversimplified "finetuning is dead" narrative and distinguishes between resource tiers.
The weakest point is the characterisation of long prompts as a direct replacement for fine-tuning,
which conflates behaviour-shaping (where long prompts work well) with knowledge injection and
representation learning (where they do not).

**Core points**

- For most teams, fine-tuning is being supplanted by long system prompts and in-context learning —
  not because fine-tuning doesn't work, but because the operational overhead outweighs the gain
  for general-purpose workflows.
- Elite teams with the infrastructure to do RLFT correctly (Cursor, Cognition) are moving in the
  opposite direction: more fine-tuning, not less.
- P/D disaggregation in inference is lowering the cost of very long context, which underpins
  the "long prompt" path's viability.
- Continued pretraining (domain-specific data throughout the training process, without a
  discrete fine-tuning stage) is a third path Jeremy Howard advocates.
- Base model capability + prompt engineering is adequate for most automation workflows; RLFT is a
  specialised capability requiring serious ML infrastructure investment.

**Impact:** For someone designing AI-integrated systems, this shapes the practical question of
how to customise model behaviour. The default answer is settling into: system prompt + RAG +
well-structured tool definitions. That's a simpler operational model with lower ongoing cost.
Reserve fine-tuning for cases where task-specific benchmark performance is a hard requirement and
you have the RL infrastructure to do it without introducing new failure modes.

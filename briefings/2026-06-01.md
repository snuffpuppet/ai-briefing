### How we contain Claude across products

**Source:** Anthropic Engineering · **Published:** 2026-05-28 · **Read time:** ~8 min

**Elevator pitch** — Anthropic documents the OS-level and VM-based containment strategies used across Claude.ai, Claude Code, and Claude Cowork, disclosing specific vulnerability incidents and internal red team results.

**Opinion vs. fact**
- [FACT] Claude.ai runs agents inside gVisor
- [FACT] Claude Code uses Seatbelt on macOS and Bubblewrap on Linux to sandbox bash commands and child processes
- [FACT] Claude Cowork runs agents inside full VMs (Apple Virtualization framework on macOS, HCS on Windows)
- [FACT] Three responsible disclosure vulnerabilities in Claude Code were reported between mid-2025 and January 2026, all exploiting code that executes before user consent is obtained
- [FACT] In a February 2026 controlled red team exercise, a researcher phished an employee into launching Claude Code with a malicious prompt; Claude completed the data exfiltration on 24 of 25 retries
- [FACT] Five distinct isolation tiers exist in production as of May 2026
- [OPINION] Requiring approval for every agent action was counterproductive due to approval fatigue
- [OPINION] Supervising what an agent is able to do is preferable to supervising what it does

**References** — Authors: Max McGuinness, Mikaela Grace, Jiri De Jonghe, Jake Eaton, Abel Ribbink (Anthropic). No external studies cited beyond internal red team results.

**Justifications** — The strongest evidence is the red team number (24/25 exfiltration successes) — concrete, internally measured, voluntarily disclosed. The vulnerability timeline (mid-2025 to Jan 2026) is asserted without severity specifics, which is a gap. The claim about approval fatigue is empirically reasonable given how Claude Code works but is argued from pattern rather than measurement.

**Core points**
- The containment philosophy is "reduce attack surface" rather than "detect bad actions": deny network egress by default, restrict filesystem writes to the workspace, sandbox at the process or VM level
- Each product sits at a different isolation tier because the threat model differs — Claude.ai (web, gVisor) vs. Claude Code (developer machine, Seatbelt/Bubblewrap) vs. Cowork (collaborative, full VM)
- Pre-user-consent code in Claude Code was the common vulnerability vector in all three disclosed bugs — patched, but a design signal for anyone building their own Claude Code integrations
- The 24/25 red team result shows that containment and social engineering bypasses coexist; sandboxes are not a complete security story
- Approval fatigue is documented as a real failure mode: users trained to click "allow" reflexively, inverting the intent of the oversight mechanism

**Impact** — Anyone deploying Claude Code in a team or enterprise context should audit whether pre-consent hooks are patched and whether their deployment tier matches their threat model. The per-product isolation matrix (gVisor → Seatbelt/Bubblewrap → full VM) is directly reusable as a reference architecture for multi-tenant agentic systems. The red team result is useful calibration: sandbox your agents, and also model the scenario where users are socially engineered into weakening those sandboxes.

### Welcome NVIDIA Cosmos 3: The First Open Omni-model for Physical AI Reasoning and Action

**Source:** NVIDIA / Hugging Face Blog · **Published:** 2026-06-01 · **Read time:** ~6 min

**Elevator pitch** — NVIDIA open-sources Cosmos 3, a single Mixture-of-Transformers model that replaces the four separate Cosmos models previously required to cover physical AI pipelines: world generation, controlled generation, scene understanding, and policy generation.

**Opinion vs. fact**
- [FACT] Cosmos 3 is built on a Mixture-of-Transformers (MoT) architecture
- [FACT] It unifies the capabilities of Cosmos Predict, Cosmos Transfer, Cosmos Reason, and Cosmos Policy into a single model
- [FACT] The model handles text, image, video, ambient sound, and action generation natively
- [FACT] Models, training scripts, and deployment tools are open-sourced on Hugging Face
- [FACT] NVIDIA formed the Cosmos Coalition, including Agile Robots, Black Forest Labs, Generalist, LTX, Runway, and Skild AI
- [OPINION] Cosmos 3 will help robots and autonomous vehicles generalize better with limited real-world training data
- [OPINION] Unifying physical AI into a single model meaningfully simplifies developer workflows

**References** — NVIDIA internal development. Cosmos Coalition partners named. No external benchmark comparisons cited in available content.

**Justifications** — The consolidation claim (4 models → 1) is directly verifiable against the prior Cosmos product lineup. The claim about improved generalization with limited data is plausible at this model scale and architecture but no published benchmark comparisons were available at publication. The open-source release on Hugging Face is verifiable. The Cosmos Coalition is a credible signal of downstream commercial interest.

**Core points**
- The MoT architecture routes computation across specialized sub-networks per modality, allowing efficient joint generation across text, video, and action without a monolithic shared backbone
- Previous Cosmos usage required stitching four separate model APIs into a pipeline; Cosmos 3 collapses this into a single model
- Primary applications: synthetic training data generation for robots and AVs, and physical AI policy model development
- NVIDIA is positioning this as both a foundation model and a fine-tuning base — the release includes training scripts, not just weights
- The Cosmos Coalition (two generative video labs, two robotics companies) signals commercial validation from builders who will use this in production

**Impact** — For teams building simulation-to-real pipelines for robotics or autonomous systems, replacing a 4-model ensemble with a single MoT model reduces operational complexity and inference surface area. The open-source release with training scripts meaningfully lowers the entry point for synthetic data generation in physical AI — previously requiring expensive closed APIs or custom pipelines. The direct telecom OSS intersection is limited, but the MoT architecture pattern — unified multimodal reasoning and action generation in one model — is a relevant design precedent for future multi-modal agentic systems.

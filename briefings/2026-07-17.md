**Elevator pitch** — OpenAI confirmed that GPT-5.6 Codex has deleted users' home directories when run in full access mode without sandboxing; the model attempts to set a temp directory by overriding `$HOME`, then mistakenly deletes it.

- [FACT] Reported file deletions have occurred when full access mode is enabled and Codex runs without sandbox protections, including without auto-review enabled.
- [FACT] The failure sequence: model attempts to override `$HOME` to define a temp directory → model makes an error and deletes `$HOME` instead.
- [FACT] OpenAI engineer Thibault Sottiaux confirmed the pattern publicly.
- [OPINION] Describing this as an "honest mistake" by the model (Sottiaux's framing) is accurate from an agent-behavior perspective but does not change the severity of the outcome.

**References** — Thibault Sottiaux (OpenAI engineer), public statement. No external citations.

**Justifications** — The failure mechanism is internally consistent and confirmed by an OpenAI engineer. What conditions prevent recurrence (model patch? sandbox enforcement?) are absent from the public statement.

**Core points**
- "Full access mode + no sandbox + GPT-5.6" is a documented configuration that can result in home directory deletion.
- The failure pattern—model misidentifies `$HOME` as a temp dir target—is a specific class of agentic error: correct reasoning about what to do, wrong identification of target.
- Auto-review being disabled is a listed contributing condition; this is a human configuration choice, not solely a model behavior.
- No model patch or fix timeline disclosed at publish time.

**Impact** — This failure mode belongs in the threat model for any agentic coding workflow running with filesystem write access. The specific condition (full access + no sandbox) should be treated as an unsafe default, not a user error. If deploying Codex-based agents, sandboxing is not an optional performance concern—it is the mechanism that prevents the agent from deleting the host environment. Review your Codex deployment configuration before the next run.

### The Archaeologist's Copilot: evidence-grounded LLMs for legacy code modernization

Source: Martin Fowler (martinfowler.com) · Published: 2026-07-16 · Read time: ~8 min

**Elevator pitch** — A first-person account of using LLMs to modernize a Java 1.5 codebase to run on modern JDKs; the central finding is that early LLM output was plausible but wrong, and progress required grounding every AI output in evidence from the actual codebase before acting on it.

**Opinion vs. fact**
- [FACT] The target codebase was Java 1.5-era code that no longer built reliably on modern hardware.
- [FACT] By 2026, Gradle 8 and modern JDKs strictly enforce package-private encapsulation that 2008-era legacy code routinely violated by accessing package-private classes across packages.
- [FACT] Early LLM-generated answers were plausible-sounding but did not hold up when applied to the actual codebase.
- [FACT] A stable Docker environment served as the validation oracle throughout; progress required Docker-confirmed correctness at each step.
- [OPINION] The author (and Fowler) conclude that AI is most useful when constrained by evidence, clear roles, and a step-by-step strategy—consistent with broader practitioner consensus but drawn from a single case study.
- [OPINION] Fowler and colleagues' CodeConcise tool (LLM + AST-derived knowledge graph) is presented as a generalization, but the article is primarily a practitioner account, not an evaluation of CodeConcise.

**References** — Nik Malykhin (case study author). Martin Fowler (editor and collaborator). Docker (stable validation environment). No external academic citations.

**Justifications** — Strongest: the core finding (early LLM answers were wrong; progress required evidence-grounding) is empirically derived from a real modernization project with a stable validation environment—not a theoretical claim. Weakest: single case study with a specific legacy context; generalizability to other stacks requires caution.

**Core points**
- LLMs confidently produce wrong answers for legacy codebases they have not been trained on; the first plausible answer should be assumed incorrect until validated.
- A stable Docker environment as the validation oracle is what makes iterative AI-assisted modernization feasible; the test harness, not the LLM, provides ground truth.
- The workflow is: use AI to analyze → validate in Docker → use AI to refactor → repeat. AI operates under human-defined evidence constraints at each step.
- Gradle 8 + modern JDK strictness on encapsulation is a concrete modernization blocker for 2000s-era Java codebases.
- CodeConcise (LLM + AST knowledge graph) is Fowler's direction for industrializing this pattern; worth tracking for teams running large legacy modernization programs.

**Impact** — For solutions architects managing legacy modernization, this is the clearest available account of what AI-assisted modernization looks like in practice: not a lift-and-shift prompt, but a disciplined evidence-gathering loop. The Docker-as-oracle approach is directly transferable. The key architecture decision is not which model to use—it is building a workflow that catches the expected default (plausible-but-wrong LLM output) before it causes regression.

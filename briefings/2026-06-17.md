### 1. Building Reliable Agentic AI Systems

**Source:** Martin Fowler · **Published:** 2026-06-16 · **Read time:** ~15 min
**URL:** https://martinfowler.com/articles/reliable-llm-bayer.html

**Elevator pitch:** Thoughtworks documents how Bayer AG evolved a pharmaceutical research assistant from keyword search into a multi-agent system that answers complex regulatory questions and drafts documents across decades of safety study PDFs.

**Opinion vs. fact:**
- [FACT] The PRINCE system (Preclinical Information Center) uses Agentic RAG and Text-to-SQL to query Bayer's safety study report archive.
- [FACT] The system can now draft regulatory documents, not just answer queries.
- [FACT] The system evolved through multiple generations, starting as keyword-based search.
- [OPINION] "Context engineering" (shaping and routing information between agents) and "harness engineering" (orchestration, recovery, observability) are the two principal design axes for reliable agentic systems.
- [OPINION] Transparency, explainability, and human-in-the-loop checkpoints are the primary trust mechanisms in regulated deployments.

**References:** Sarang Sanjay Kulkarni (Thoughtworks). Bayer AG as project client. No external studies cited.

**Justifications:** The article draws on a delivered production system, which is stronger evidence than theoretical architecture proposals. The "context vs. harness engineering" framing is a useful conceptual split but is an organizing principle derived from practice, not a formally validated model. The trust-via-transparency argument is asserted based on user feedback, not measured systematically.

**Core points:**
- Agentic RAG combines unstructured PDF retrieval with Text-to-SQL against tabular data — necessary when the knowledge base is heterogeneous and decades old.
- Context engineering operates at the system level: which information reaches each agent, in what form, filtered how. Comparable to prompt engineering but for data pipelines.
- Harness engineering covers orchestration, error recovery, and observability — the infrastructure that prevents agentic pipelines from failing silently on unexpected inputs.
- Human-in-the-loop checkpoints are a designed feature in regulated domains, not a concession to model limitations.
- The system moved from search to synthesis: capable of generating regulatory document drafts, not just surfacing citations.

**Impact:** This is a worked example in a domain with strict quality requirements — regulated pharma — making it more transferable than most AI case studies. The context/harness engineering split is directly applicable to any multi-agent system with tool use and chained LLM calls. The harness engineering emphasis addresses the most common production failure mode in agentic pipelines: silent degradation under inputs that fall outside the training distribution. For someone designing AI-integrated OSS platforms, the observability and recovery architecture described here is the relevant takeaway.

### 3. Critical Copilot Vulnerability Allowed Hackers to Steal 2FA Codes from Users

**Source:** Ars Technica · **Published:** 2026-06-16 · **Read time:** ~5 min
**URL:** https://arstechnica.com/security/2026/06/critical-copilot-vulnerability-allowed-hackers-to-seal-2fa-code-from-users/

**Elevator pitch:** A critical flaw in Microsoft 365 Copilot combined prompt injection, server-side request forgery, and an HTML rendering race condition to extract authentication codes, emails, and confidential files via a single user click.

**Opinion vs. fact:**
- [FACT] CVE-2026-42824, rated critical.
- [FACT] The attack chain (dubbed "SearchLeak") combines Parameter-to-Prompt (P2P) injection, an HTML rendering race condition, and an SSRF via Bing.
- [FACT] A single click triggers Copilot to search enterprise data sources and generate a response with hidden exfiltration payloads — no further user interaction required.
- [FACT] Data accessible to the exploit included OTPs/2FA codes, email content, calendar events, SharePoint files, OneDrive documents, salary data, and acquisition plans.
- [FACT] Microsoft patched it via a cloud backend update; no action required from IT admins.
- [FACT] The fix introduced "AI Request Verification" — a mandatory user-facing confirmation dialog when Copilot is about to send data to an external endpoint.
- [OPINION] This vulnerability shows why the AI industry's current approach to LLM security fails repeatedly (Ars Technica framing).

**References:** CVE-2026-42824; Microsoft security response; multiple security vendors including GBHackers, The Cyber Express, AndroidHeadlines, Aviatrix.

**Justifications:** The attack chain is well-documented across multiple independent security vendors, and Microsoft confirmed and patched it. The "industry keeps failing" editorial framing is consistent with the documented history of structurally identical prompt injection exploits in other LLM products (including EchoLeak in 2025), lending it empirical support beyond opinion.

**Core points:**
- P2P injection differs from classic prompt injection: the attacker controls a URL parameter that gets reflected into Copilot's internal prompt, bypassing input-visible injection detection.
- The SSRF component allowed Copilot to issue outbound requests to attacker-controlled endpoints, bypassing normal egress controls.
- The vulnerability existed because Copilot's data access (email, SharePoint, live OTPs) was broad and no per-request egress verification existed.
- Microsoft's "AI Request Verification" adds a user-facing confirmation step before external sends — a defense-in-depth measure that doesn't fix the injection root cause.
- LLMs with broad internal data access plus uncontrolled outbound connection capability constitute a structurally exploitable attack surface.

**Impact:** This is a template vulnerability for the class of exploits that emerges when LLMs have both read access to sensitive data and the ability to make outbound connections. Any agentic system designed with similar access patterns — an agent that reads internal data and calls external APIs — needs explicit, auditable egress controls at the infrastructure level, not just prompt-level input filtering. The AI Request Verification pattern (confirm before external send) is a directly implementable safeguard. For enterprise Copilot deployments: patched, no action needed. For architects building similar systems: model this threat explicitly.

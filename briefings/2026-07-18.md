### In-House LLM Serving at Netflix

Source: Netflix TechBlog · Published: 2026-07-17 · Read time: ~15 min

**Elevator pitch:** Netflix explains why they run the entire LLM serving stack in-house (vLLM + Triton rather than managed APIs), and which production trade-offs only became visible under live load.

**Opinion vs. fact**

- [FACT] Netflix runs the full LLM serving stack in-house, within their existing production environment rather than in a separate ML silo.
- [FACT] They selected vLLM as their inference engine, choosing it over TensorRT-LLM.
- [FACT] Their deployment uses NVIDIA's Triton with the OpenAI-compatible vLLM backend; the frontend converts request schemas into Triton inference calls and serves via FastAPI.
- [FACT] Model packaging: JSON config files pointing to weights and tokenizer; Triton's vLLM backend reads these and generates I/O tensor specs dynamically.
- [FACT] Netflix serves four workload types: embedding generation, prefill-only inference (ranking/retrieval), autoregressive decoding, and custom models with per-step constraint logic.
- [FACT] They encountered Triton/vLLM version mismatch in production: Triton 25.09 imported `vllm.engine.metrics`, a module removed in vLLM 0.11.2.
- [OPINION] vLLM's extensibility and debuggability outweighed raw throughput differences vs. compiled engines for Netflix's workload mix.
- [OPINION] By mid-2025, open-source serving engines had closed the performance gap with specialized stacks, making operational characteristics the decisive criterion.

**References:** Netflix TechBlog engineering post (July 17, 2026); additional context from Netflix's May 2026 routing infrastructure post.

**Justifications:** The article's value is the specificity of the decision rationale — custom model loading without a compilation step, extensibility hooks for constraint decoding, ML practitioner familiarity with vLLM — rather than aggregate throughput numbers. The version-pinning incident (Triton 25.09 / vLLM 0.11.2 incompatibility) is a concrete production failure mode, not a hypothetical. The main caveat: Netflix's workload mix — four distinct inference modalities under one serving platform — is broader than most single-purpose deployments, so some conclusions require translation to narrower contexts.

**Core points**

- vLLM chosen over TensorRT-LLM: loads custom model architectures without a compilation pipeline; provides extensibility hooks for custom decoding logic; better debuggability; familiar to ML practitioners already using vLLM in research.
- Frontend stack: NVIDIA Triton (OpenAI-compatible endpoint) → vLLM backend; model artifacts are JSON configs pointing to HuggingFace-format weights.
- One platform, four workload types: embeddings, prefill-only (ranking/retrieval), standard autoregressive generation, and custom per-step constraint models.
- Production risk: Triton and vLLM must be version-pinned as a pair; vLLM's internal APIs (e.g., `vllm.engine.metrics`) can vanish between minor releases without public notice.
- Decision driver: by summer 2025, the performance gap between open-source serving engines and specialized stacks had closed enough that operational fit became the dominant selection criterion.

**Impact:** Netflix's post is a rare published account of an in-house LLM serving decision at production scale, with named alternatives and concrete failure modes. For architects evaluating self-hosted inference: the vLLM selection rationale is a useful decision template when workloads are heterogeneous and model architectures evolve frequently. The version-pinning problem is immediately actionable — teams should lock Triton/vLLM pairs in container image builds rather than pulling latest on each deploy. The broader lesson: operational ergonomics (debuggability, customizability, developer familiarity) now often outweigh raw throughput differences in production serving selection.

**Justifications:** The parallelism-as-config design is the architectural claim most worth evaluating. Supporting references in NeMo's documentation confirm this is an actual design property, not a marketing description. **Core points**
